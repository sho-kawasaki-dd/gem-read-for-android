# **企画書: Gem Read for Android**

ブラウザ上のテキストや画面上の画像（スクリーンショット）をシームレスに取得し、Gemini APIを用いて翻訳・解説を行うAndroid向けユーティリティアプリ。PC版「Gem Read (pdf\_epub\_reader)」の姉妹プロジェクトとして、独立したリポジトリで開発する。

## **0\. プロジェクトの目的とゴール**

* **ゴール:** Android端末上での読書やブラウジング中に発生する「これなんて書いてある？」「この図表の意味は？」という疑問を、最小の手数でGeminiに投げ、結果を得られるようにする。  
* **ターゲット:**  
  * 第1段階: 開発者自身（日々の情報収集の効率化）  
  * 第2段階: OSSとして公開し、海外の論文や技術記事を読むエンジニア・学生を支援する。

## **1\. 開発フェーズとロードマップ**

OSSとしての保守性と確実なリリースを両立するため、手堅いアプローチから段階的に開発を進める。

### **Phase 1: OS標準連携の確立（MVP: Minimum Viable Product）**

自前での画面キャプチャは行わず、Android OSが提供する標準のインテント（共有・テキスト選択）を受け取る「受け口」としてアプリを構築する。

* **機能要件:**  
  1. **テキスト共有の受信:** ブラウザ等からの ACTION\_SEND (text/plain) を受け取る。  
  2. **画像共有の受信:** OS標準機能で撮影・トリミングされたスクショの ACTION\_SEND (image/\*) を受け取る。  
  3. **テキスト選択メニューの拡張:** テキスト選択時のポップアップメニューに「翻訳(Gem Read)」を追加する（ACTION\_PROCESS\_TEXT）。  
  4. **Gemini API連携:** 受け取ったテキスト/画像をGemini API (Flash または Pro) に送信し、結果を表示する。  
  5. **設定画面:** APIキーの登録、使用モデルの選択、ベースとなるシステムプロンプトの編集。

### **Phase 2: オーバーレイ＆シームレスキャプチャ（Advanced UX）**

ユーザーの手数を極限まで減らすため、自前での画面キャプチャと矩形選択UIを実装する。

* **機能要件:**  
  1. **フローティングトリガー:** 他のアプリの上に常駐するボタンを表示（SYSTEM\_ALERT\_WINDOW）。  
  2. **セッション型キャプチャ:** MediaProjection APIとForeground Serviceを組み合わせ、Android 14+の録画許可ダイアログを「セッション開始時の1回」に抑える。  
  3. **矩形選択UI:** 画面全体を暗くするオーバーレイを展開し、指でなぞった範囲の画像をBitmapとして切り出してGeminiに送信する。  
  4. **フォールバック:** セッションがOSにキルされた際の再認証フロー。

## **2\. 技術スタック（推奨案）**

OSS化と将来のメンテナンス性を考慮し、現在のAndroid開発におけるモダンな標準構成を採用する。

* **言語:** Kotlin  
* **UIフレームワーク:** **Jetpack Compose**  
  *理由: 従来のXMLベースのレイアウトに比べ、状態管理（APIのロード中、成功、エラー等）の記述が圧倒的にシンプルになり、UIコンポーネントの再利用性が高いため。*  
* **アーキテクチャ:** MVVM (Model-View-ViewModel) \+ クリーンアーキテクチャの簡易版  
* **非同期処理:** Kotlin Coroutines & StateFlow  
* **API通信:** Google公式の google-genai Android SDK（またはKtorを用いたRESTクライアント）  
* **ローカル保存:** Jetpack DataStore (Preferences) ※旧SharedPreferencesの代替。APIキーなどの保存に使用。

## **3\. UI/UX フローの設計方針（Phase 1時点）**

インテント経由でアプリが呼び出された際の「結果表示画面」のUIをどうするかについて、2つのアプローチがある。

* **A案: フルススクリーンActivity**  
  通常のアプリ起動のように全画面を占有する。文字は読みやすいが、元のブラウザ等の文脈から完全に切り離される。  
* **B案: Bottom Sheet型 Activity（推奨）**  
  画面の下半分だけにニュッとUIがせり出してくる形式（ダイアログテーマのActivity）。  
  *理由: ユーザーが「元の記事を読みながら、下半分で翻訳結果を確認する」という並行作業が可能になり、UXが劇的に向上するため。*

## **4\. 必要なパーミッション（権限）**

**Phase 1:**

* android.permission.INTERNET (通信用)

**Phase 2 で追加:**

* android.permission.SYSTEM\_ALERT\_WINDOW (フローティングUI用)  
* android.permission.FOREGROUND\_SERVICE  
* android.permission.FOREGROUND\_SERVICE\_MEDIA\_PROJECTION (画面キャプチャ維持用)

## **5\. オープンな論点（設計時に決めるべきこと）**

1. **APIキーのセキュリティ:**  
   ユーザー自身のAPIキーを入力させる設計になるが、DataStoreに平文で保存するか、Android Keystoreを使って暗号化して保存するか。（*推奨: OSS公開を前提とするなら、実装はやや複雑になるがKeystoreによる暗号化保存を最初から組み込むべき*）。  
2. **PC版とのプロンプト同期:**  
   「定型文」をどのように管理するか。アプリ内にハードコードするのではなく、ユーザーが自由に編集・保存できる機能（プロンプトテンプレート機能）がPhase 1の段階から必要か？  
3. **画像送信時のリサイズ:**  
   高解像度のスクショをそのままGemini APIに送ると通信量・トークン量が膨大になる。送信前にアプリ側で画像をダウンスケールするロジックを入れるか。
